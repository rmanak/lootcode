# Input-constraint validators

Every problem ships an **input-constraint validator** at:

```
content/problems/<slug>/input_validator/input_validator.py
```

It is a small, dependency-free Python module exposing one predicate:

```python
def validate_input(<the problem's params>) -> bool:
    """True iff the arguments satisfy the input constraints the statement states."""
```

The parameters are exactly the problem's `function.params` from `meta.json`, in
order and by name — the same keys used in a test case's `input` object. The
predicate returns `True` when a candidate input is **in-bounds** for the problem
(value ranges, lengths, shapes, allowed character sets, structural invariants…)
and `False` otherwise. It deliberately does **not** check un-observable *semantic*
guarantees ("exactly one answer exists", "the graph is connected") — only the
constraints a caller can verify from the input alone.

These were bulk-generated by `scripts/generate_constraint_validators.py` (an LLM
reads each statement + signature and writes the predicate), then audited so that
each one accepts all of its own known-legal test inputs.

## Why they exist — and when to use them

**The main use is adding test cases.** When you add a new `(input, expected)`
pair to a problem's `tests/cases.json`, run the input through `validate_input`
first. If it returns `False`, the input violates a constraint the statement
promises (e.g. an out-of-range value, a too-long array, a lowercase-only string
that isn't), so it is not a legal input for that problem — fix the input (or, if
the statement's constraint is genuinely wrong, fix the statement **and** the
validator together). This keeps our test cases honest: solvers are only ever
judged on inputs the problem actually allows.

They are also a cheap executable oracle for fuzzing / test-strengthening (reject
inputs the statement forbids) and for auditing that the existing bank is
in-bounds.

## The contract

- **Entry point:** a top-level function named `validate_input`. (A handful of
  older validators were authored as `validate_inputs`; they were normalized to
  the singular name during the move into the content tree.)
- **Signature:** one parameter per `meta.json` `function.params` entry, same
  names. Call it with keyword arguments taken straight from a case's `input`:
  `validate_input(**case["input"])`.
- **Pure & self-contained:** no imports of project code, no I/O. Safe to `exec`
  in an isolated namespace and call.
- **`bool` is not `int`:** validators explicitly reject `True`/`False` where an
  integer is required (Python's `bool` is a subclass of `int`).

### Rich types (`TreeNode`, `ListNode`, `DoublyLinkedList`)

Parameters whose declared type is a rich type are stored in `tests/cases.json` as
their **serialized** form: a `TreeNode` is a LeetCode-style level-order list with
`null` for absent children; a `ListNode`/`DoublyLinkedList` is a **flat list of the
node values** in order. A validator validates that stored form.

- For linked-list types the stored form is a plain value list, so the natural
  validator (list type, length bound, per-value bounds — exactly like an `int[]`)
  works as-is; no decoding needed.
- For `TreeNode` a validator may reasonably check *either* the raw list or a
  decoded node object, so the audit tooling (`verify_against_cases` in
  `generate_constraint_validators.py`) scores it under **both** encodings and
  accepts it if it passes under either.

See `app/executor/harness.py` `_CODECS` for the canonical decoding.

## Checking a problem (or the whole bank)

`scripts/check_constraint_validators.py` loads each validator and runs it over
that problem's own `tests/cases.json` inputs. Every stored input is legal by
construction, so a good validator returns `True` on all of them.

```bash
python scripts/check_constraint_validators.py                 # audit every problem
python scripts/check_constraint_validators.py --slug two-sum  # just one
python scripts/check_constraint_validators.py -v              # per-problem PASS lines
```

Non-zero exit if any validator rejects a legal input or fails to load — usable as
a CI gate. To validate one *ad-hoc* candidate input before adding it as a case:

```python
import importlib.util, pathlib
spec = importlib.util.spec_from_file_location(
    "v", "content/problems/two-sum/input_validator/input_validator.py")
mod = importlib.util.module_from_spec(spec); spec.loader.exec_module(mod)
assert mod.validate_input(nums=[2, 7, 11, 15], target=9)   # must be True
```

## Regenerating / adding a validator

`scripts/generate_constraint_validators.py` writes freshly generated validators
into a flat **staging** directory (`--out-dir`, default `constraint_validators/`)
under `<slug>_input_test.py`, or `<slug>_input_test.flagged.py` if the validator
rejected one of its own legal inputs and needs review. Audit a staging dir with
`check_constraint_validators.py <staging_dir>`. Once a staged validator looks
right, **promote** it into the content tree:

```
content/problems/<slug>/input_validator/input_validator.py   # rename fn to validate_input
```

The staging directory is scratch and is not committed; the per-problem
`input_validator/` files are the source of truth.

## Notes & limitations

- The app does **not** load these at runtime — they are an authoring/QA aid, not
  part of the judge. `app/content.py` only reads `meta.json`, `problem.md`,
  `tests/cases.json`, `starters/`, `solution/`; the `input_validator/` folder is
  ignored by loading and untouched by write-back.
- A validator encodes the constraints *as stated at generation time*. If you
  change a problem's stated constraints, update its validator too (and re-run the
  audit).
- Validators are best-effort machine output. A `True` means "no stated constraint
  is violated," not "this is a good test case."
