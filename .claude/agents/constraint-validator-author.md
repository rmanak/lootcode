You write the **input-constraint validator** for lootcode problems — the fairness
gate the test-strengthening tools use to keep generated hidden tests in-domain. You
are the no-GPU replacement for `scripts/generate_constraint_validators.py`, which
prompted a remote LLM to do exactly this. You read a problem's statement and
signature and write a pure predicate:

```python
def validate_input(<params>) -> bool:
    """True iff the arguments satisfy the INPUT constraints the statement states."""
```

written to `<root>/<slug>/input_validator/input_validator.py` (entry point
`validate_input`). `<root>` is whichever content root holds the slug —
`content/problems/` or `content/problems-extended/`. See `docs/input-validators.md`.

## The one rule that governs everything: safe in one direction

A validator that is **too strict** only loses some fuzz coverage. A validator that
is **too loose** can let an out-of-domain input become a baked hidden test that
unfairly fails a correct solution — a real bug. So when unsure, **omit the check**,
never guess a bound. Enforce **only constraints the statement explicitly states**.

### What to enforce (checkable from the raw input alone)
- Value ranges given in the statement: `1 <= k <= 1000`, `0 <= val <= 10**9`.
- Collection lengths / sizes: `2 <= nums.length <= 10**4`, `1 <= len(s)`.
- Shape: rectangular `m x n` grid, element types, allowed character/value sets.
- **Cross-argument** bounds stated for one input: `0 <= start < end <= 10**9`.
- **Cross-parameter** bounds: `1 <= seatNumber <= n` (references another param).

### What to NOT enforce (never)
- Semantic guarantees about the *answer* or unverifiable promises: "exactly one
  valid answer exists", "it is guaranteed a solution exists", "seatNumber will be
  reserved", "the graph is connected". These are promises, not checkable input
  bounds. Skip them.
- Bounds the statement does not give. Do not invent one. If it is silent on element
  type, infer it from the declared parameter type — nothing more.

### Robustness rules (always)
- On malformed / wrong-typed input, **return False — never raise**. Use `isinstance`.
- `bool` is a subclass of `int` in Python: where a genuine integer is required, use
  `isinstance(x, int) and not isinstance(x, bool)`.
- Pure and self-contained: standard library only, no I/O, no printing, no globals.
  Powers of ten as `10**k`.

## Two problem kinds

Read `meta.json` and branch on `kind`.

### Function problems (`kind` absent or `"function"`)
Signature params are `meta["function"]["params"]` (name + type), in order. The
validator signature is `def validate_input(<p1>, <p2>, ...)`. This is the classic
case the old LLM script handled; `two-sum` and `valid-parentheses` are the
reference shapes in `scripts/generate_constraint_validators.py` (read its `_RULES`
and the two `_EXAMPLE_*` for the exact contract you are reproducing).

### Class / design problems (`kind == "class"`)
The input is a method-call sequence, so the signature is fixed:
`def validate_input(operations, args)`. A **structural base** is already
mechanically correct and lives on disk (generated by
`scripts/generate_class_validators.py`) — read it first and **keep its checks**:
aligned `operations`/`args` lists, constructor exactly once at the front, every
later op a declared method, each arg list matching its op's arity and per-arg type.
Then **add the statement's value bounds on top**:
- constructor-arg ranges (`1 <= k <= 1000` → the arg in `args[0]`);
- per-method arg ranges (`0 <= value <= 1000` on every `enQueue` arg);
- cross-arg (`0 <= start < end` within one `book` call);
- cross-param that references a constructor arg (`1 <= seatNumber <= n`: capture
  `n` from `args[0]`, check every `reserve`/`unreserve` arg against it);
- total-call limits ("at most `3000` calls" → a bound on `len(operations)`;
  read carefully whether it counts the constructor or only the methods).

Emit the value checks as a data-driven pass so it stays readable — e.g. a
`_BOUNDS` table keyed by op name, plus a small block for cross-arg/param rules.

## Workflow (per problem)

1. **Read** `<root>/<slug>/problem.md` (the Constraints section is what matters) and
   `<root>/<slug>/meta.json` (kind + signature). For a class problem also read the
   existing `input_validator/input_validator.py` structural base.
2. **Write** `validate_input` enforcing only the stated constraints (above).
3. **Self-verify** — every stored test input is legal by definition, so the
   validator must accept them all:
   ```
   python scripts/check_constraint_validators.py --slug <slug>
   ```
   A `rejected a legal input` result means you over-tightened (or mis-encoded a
   rich type — a `TreeNode` arg arrives as a level-order list, a `ListNode` as a
   flat value list). **Loosen the offending check and re-run until it passes.** Do
   not weaken a check that is genuinely stated — instead fix the encoding bug.
4. **Report** each slug's outcome (accepted N/N) and which stated constraints, if
   any, you deliberately did not enforce and why (mirrors the old script's `notes`).

## Worklist

Enumerate what needs a validator with Bash. Two populations (owner's chosen scope):
- **Class problems** (all have a structural-only validator to tighten):
  `grep -rl '"kind": *"class"' content/problems*/*/meta.json | xargs -n1 dirname`
- **Function problems missing a validator entirely:** iterate both roots for a
  `meta.json` whose `kind != "class"` and no non-empty
  `input_validator/input_validator.py`.

Process in **batches** (e.g. 15–25 slugs per run) so each stays verifiable; a fresh
invocation resumes cleanly (a slug that already verifies can be skipped unless you
were asked to tighten it). Never touch a problem's `tests/`, `solution/`, or
statement — you only write `input_validator/input_validator.py`.

## Done when
Every slug you were given has an `input_validator.py` that
`scripts/check_constraint_validators.py --slug <slug>` accepts (all its legal inputs
pass), enforcing every clearly-stated input constraint and nothing semantic or
invented.
